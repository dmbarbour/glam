# Code and Configuration Review — 2026-07-16

## Scope and priorities

This is a read-only review of the Rust bootstrap after removal of the core
expression layer. It does not propose semantic changes or include implementation
edits. Priorities are:

1. clarity of names, ownership boundaries, and comments;
2. executable-size reduction, guided by measurements;
3. low-risk performance improvements that do not pre-empt later architecture.

Required design references were read first: `DistilledDesign.md`,
`AgentContext.md`, `src/README.md`, `Overview.md`, and `DesignPrinciples.md`.

## Review method

- Map modules, public APIs, large functions, repeated patterns, and dependency
  ownership.
- Compare implementation comments and agent documentation with current code.
- Build and measure the `release-small` binary, then attribute size using the
  tools already available in the workspace.
- Inspect common allocation, cloning, hashing, locking, and traversal paths for
  low-hanging improvements.
- Rank recommendations by clarity benefit, confidence, risk, and likely effort.

## Executive summary

The codebase is in a much better architectural position than its file sizes and
documentation currently suggest: syntax owns lambdas and resolution, core owns
values, and the generic interaction-net runtime is separated from the core
specialization. The highest-value work now is consolidation rather than another
semantic transition.

The main review findings are:

- `g_syntax.rs`, `eval.rs`, and `interaction_net.rs` are each carrying several
  distinct responsibilities. Splitting them along existing conceptual seams is
  the clearest way to make the implementation navigable without changing its
  semantics.
- Documentation still narrates several completed spikes. `AgentContext.md` and
  `src/README.md` duplicate detailed interaction-net history, while some of
  their simpler current-state claims have gone stale.
- A small amount of compatibility surface survived the expression removal even
  though nothing constructs or calls it. Removing that surface would make the
  current model easier to state and test.
- The 882 KiB executable is not caused by a missing release flag. The current
  profile already uses the important stable size settings. Parser
  monomorphization and evaluator dispatch are the largest application-owned
  areas; the generic interaction-net reducer is not the main size problem.
- There are useful allocation and hashing experiments available, but no
  performance issue found in this pass justifies disrupting the current
  interaction-net scheduler. Clarity should remain the immediate priority.

## Clarity and maintainability

### Documentation ownership

`docs/AgentContext.md` is no longer compact. Its evaluator section contains a
long chronological account of the interaction-net spike, including detailed
cursor algorithms and temporary compatibility decisions. `src/README.md`
repeats much of the same material as one very dense architecture walkthrough.
Both are useful source material, but neither currently serves its stated role
well. Establish one short current-state module map and keep detailed
interaction-net invariants in the existing focused agent-context document.

There are also concrete stale statements:

- `AgentContext.md` names `src/numbers.rs`; the module is `src/number.rs`. (FIXED)
- Its command-line section says the current spike accepts one `--file`, while
  `main.rs` also handles scripts, configuration inputs, `--parse`, `--`, and
  multiple assembly inputs.
- Its generic `parser`, `data types`, and `evaluator` TODOs no longer identify
  actionable work.
- `docs/agent_context/object_spike.md` contains both a short plan and a pasted,
  older full plan. Much of it is chronological rather than a current invariant.

Recommended ownership:

- Keep `docs/AgentContext.md` to non-obvious invariants, common regressions, and
  links to focused notes. Its long `Eval` section should become a short boundary
  summary plus a link.
- Make `src/README.md` the current module map and end-to-end dataflow, not a
  second interaction-net design document.
- Rewrite `docs/agent_context/interaction_net_spike.md` as the authoritative
  current interaction-net invariants document; once it no longer describes a
  temporary spike, rename it accordingly.
- Either rewrite `object_spike.md` to only the still-current object invariants,
  or move the historical plan under `docs/tmp/`.
- Keep target-state documents such as `DistilledDesign.md` distinct from these
  current-implementation documents. Do not make bootstrap behavior sound like
  settled language design.

### Module boundaries

The three largest source files contain natural internal boundaries already:

- `g_syntax.rs` is 8,598 lines and combines source declarations, parsing,
  diagnostics, resolution, capture analysis, and interaction-net emission.
  Use the existing `g_syntax/resolved.rs` submodule as the seed of the internal
  structure, then move resolved-net emission and parser/declaration handling to
  adjacent submodules. Keep syntax-to-semantic lowering in `g_syntax`; this is
  a file organization change, not permission to leak syntax into `core`.
    - (Implemented)
- `eval.rs` is 4,898 lines and combines value forcing, net-interface driving,
  operator application, builtin semantics, access/object compatibility, and
  tests. `apply_builtin` alone is approximately 31 KiB of machine code in the
  measured build. Split builtin implementations by semantic family behind one
  small dispatcher, and separate shared-net driving from core value forcing.
- `interaction_net.rs` is 3,272 lines and combines the immutable template and
  builder, runtime storage/scheduling, cursor copying, rewrite rules, and tests.
  The generic boundary is good; preserve it. Internal submodules for
  construction, runtime/cursors, and reduction would expose that structure
  without changing the public abstraction.

These totals include substantial inline tests: the main production regions are
roughly 5.3K, 3.3K, and 2.1K lines respectively. Moving large test fixtures to
named test submodules will improve navigation but will not reduce the release
binary. In particular, `eval.rs` currently starts with a test-only expression
evaluator before its production entry point, which makes the removed core
expression model appear more current than it is.

Do these splits incrementally and keep tests beside the owning behavior. A
large mechanical module move would create review noise without improving the
model.

#### Clarification

**Q:** Seed of internal structure? Do you just mean more submodules?

Yes—“seed of the internal structure” mostly means more private submodules under `g_syntax`, using `resolved.rs` as the existing precedent. I did not mean a new architectural layer or abstraction.

For example:

```text
src/
├── g_syntax.rs              # Public facade and pipeline orchestration
└── g_syntax/
    ├── ast.rs               # SyntaxExpr, declarations, parsed syntax types
    ├── parser.rs            # Chumsky parsers and source normalization
    ├── resolve.rs           # Names/scopes; SyntaxExpr -> ResolvedExpr
    ├── resolved.rs          # BindingId, ResolvedExpr, free-binding analysis
    ├── net_lowering.rs      # ResolvedExpr -> closed interaction nets/Value
    └── module_lowering.rs   # Imports, definitions, objects, module assembly
```

That is an eventual shape, not a recommendation to create all those files immediately. A narrower first split would be:

1. Keep `BindingId`, `ResolvedExpr`, and `free_bindings` in `resolved.rs`.
2. Move `ResolvedNetLowerer` and `lower_resolved_expr` into `net_lowering.rs`.
3. Move the Chumsky expression-parser construction into `parser.rs`.
4. Leave the remaining declarations and orchestration in `g_syntax.rs` until another boundary becomes obvious.

Then `g_syntax.rs` acts as the facade:

```rust
mod net_lowering;
mod parser;
mod resolved;

pub struct SourceFile { /* ... */ }
pub struct ParsedSource { /* ... */ }

pub fn parse_source(/* ... */) -> ParsedSource { /* ... */ }
pub fn lower_source(/* ... */) -> LoweredSource { /* ... */ }
```

The important ownership direction remains:

```text
source syntax
    ↓ parser
SyntaxExpr
    ↓ resolver
ResolvedExpr
    ↓ net lowering
Value / SharedRuntimeNet
```

All of those stages belong to the `g_syntax` front end. `core` should know only about the resulting values and specialized net data—not `SyntaxExpr`, lexical names, captures, lambdas, or parsing.

So “seed” was slightly grandiose wording. More precisely: use the existing submodule as the starting point for a few responsibility-based, private submodules, while keeping `g_syntax` itself as one front-end boundary.

### Leftover and redundant surface (Concrete removals implemented)

The concrete dead APIs below have now been removed.
`CoreSpecialization::Data` is `Value` directly, and the release-small
executable decreased from 903,112 to 901,832 bytes. Partitioning the broader
public API remains a separate design task.

At review time, the following items had no producer or consumer outside their
own match arms or definitions:

- `CoreNetData::Builtin` has no constructor. Builtins now enter nets through
  operators, so this variant preserves an obsolete second representation.
- `CoreOperator::Error` has no constructor. If errors are represented by ready
  failed lazy values and ordinary stuck operator outcomes, this variant should
  be removed; otherwise add the missing single documented construction path.
- `RuntimeNet::take_interface_data` has no caller.
- `LazyValue::get` is only used by `LazyValue::cached` and exposes the same
  operation twice.

Removing these is preferable to adding comments that explain paths the program
cannot take. Re-run the search immediately before editing because these are
public APIs and Rust's ordinary dead-code lint does not prove they are unused by
external crates.

The removal was followed through instead of retaining a one-variant wrapper:
the core specialization's `Data` is now `Value` directly. This eliminated the
repeated `CoreNetData::Value(...)` wrapping and its conversion match layers.

The crate also exports every module wholesale. That is understandable during a
spike, but it makes implementation helpers look like supported library API and
prevents dead-code lints from helping much. Before adding more construction
features, distinguish the intended public front-end surface (`CompileContext`,
values, and checked net construction) from evaluator/runtime internals. Keep
`NetBuilder`'s generic construction API public if it is deliberately reusable;
most scheduling and claimed-pair mutation methods can remain crate-owned even
though their types are generic.

### Naming and comments

Recommended focused renames and comment edits:

- Add short module-level ownership docs to `compiler`, `core`, `eval`, and
  `g_syntax`, matching the useful module docs already present in `core_net`,
  `interaction_net`, and `list`. Each should state what the module owns and one
  important thing it deliberately does not own. This is the best location for
  the syntax/core/runtime boundary; do not repeat it above many individual
  functions.
- `lower_to_core_with_context` still evokes the removed `CoreExpr` lowering
  phase. `lower_source` or `compile_source` better describes producing a core
  value/net from resolved syntax. Rename `LoweredSource` consistently if it is
  the compiled module result.
- Variables named `term` in `main.rs` are lambda-era leftovers; use
  `module_value`, `definitions`, or the exact role.
- `instantiate_module` primarily fulfills the final-definition cell and returns
  the module definitions. A name such as `fulfill_final_definitions` or
  `seal_module` would state that lifecycle better than “instantiate.”
- `FunctionCode` contains a shared interaction-net runtime rather than code in
  a general sense. `FunctionNet` or `FunctionTemplate` would make its
  representation explicit. Choose the latter only if it really remains an
  immutable reusable template; today it is a shared runtime stage.
- `LazyValue::{get,set,cache,cached}` obscures the distinction between peeking,
  one-time IVar fulfillment, and memoization. Prefer lifecycle names such as
  `peek_result`, `fulfill`, and `memoize`, with one public peek operation.
- `CoreOperator::{FunctionCaptures,ComputationCaptures}` accumulate staged
  inputs. `CollectFunctionCaptures` and `CollectComputationCaptures` state that
  behavior more directly.
- Most `CoreOperator` variants repeat the same “metadata plus supplied
  arguments” protocol. A small `StagedArgs` helper could own the append,
  remaining-arity, and completion checks for apply, captures, dict/list
  construction, builtin calls, and access. This would remove repeated
  `Arc<[Value]> -> Vec<Value> -> Arc<[Value]>` boilerplate and give later
  performance work one representation to improve. Do not obscure the distinct
  operator meanings behind a generic framework; centralize only their shared
  staging lifecycle.
- The `BindingId` comment in `g_syntax/resolved.rs` still contrasts the ID with
  a removed “Core compatibility backend.” Replace it with the current
  invariant: identity is context-local, stable during one resolution/lowering
  pass, and never globally allocated.
- The comment on `LoweredSource::definitions` still describes an open lambda
  fixpoint. State the actual value/cell protocol instead.
- Convert the fragmentary `Atom` comments into one short doc comment explaining
  that it is the interned representation of `[Key]:()` and therefore has cheap
  equality and hashing.
- `SharedRuntimeNet::{with,with_mut}` execute a closure while holding its mutex.
  Names such as `inspect` and `update` (or explicitly `with_locked` and
  `with_locked_mut`) would make the critical-section boundary more visible at
  call sites.
- Several test names and helpers in `eval.rs` still say “core terms,” “lambda
  evaluator,” and “closures.” Rename them to describe the source construct and
  resulting net/function behavior. Keeping a test-only expression fixture is
  fine; making it sound like production core IR is not.

Do not mechanically expand the domain names `defs` and `deps`: they match the
language's object-spec keys and are clearer than longer aliases in that scope.
Likewise, do not shorten useful source diagnostics to save string bytes.

### CLI and compiler-context cleanup

- `main.rs` repeats file/script argument parsing and diagnostic emission. Small
  helpers for parsing one assembly input, reading a `&Path`, and printing a
  diagnostic group would reduce branching without introducing a large CLI
  dependency.
- Configuration discovery accepts both `GLAS_CONF` and `GLAM_CONF`, and uses
  `glas` in platform default directories. Decide whether this is intentional
  compatibility. If not, use `glam` consistently; if so, document the legacy
  name and its precedence.
- `CompileContext` mixes public fields with builder/accessor methods. Choose one
  model: an explicit capability record with documented fields, or an
  encapsulated API. The current halfway point makes ownership harder to infer.
- `SourceFile` retains a `String` while `CompileContext::with_source_binary`
  copies its bytes into another owner. Establish one source-byte owner and make
  UTF-8 parsing borrow from it. This simplifies the API and avoids a full-source
  copy.
- Tiny `CompileContext::value_*` constructors are reasonable if they are the
  deliberate front-end effects boundary. Document that purpose rather than
  removing them merely because their current implementations are trivial.

### Small correctness issue found during the audit

`number.rs` converts exponent magnitudes through unchecked `i64 -> usize -> u32`
casts before `pow`. The first conversion can truncate on 32-bit targets and
the second can truncate on every target; even before truncation, an enormous
accepted exponent is an uncontrolled allocation request. Define a supported
exponent bound, convert it explicitly, and return the existing numeric error
when it is exceeded. This is a correctness/robustness cleanup, not a performance
task.

## Size measurements

The baseline `cargo build --profile release-small` executable is 903,112 bytes,
or 881.95 KiB. The relevant ELF sections are:

| Section | Bytes |
| --- | ---: |
| `.text` | 653,799 |
| `.rodata` | 67,792 |
| `.eh_frame` | 110,048 |
| `.eh_frame_hdr` | 15,252 |
| `.rela.dyn` | 22,824 |
| `.gcc_except_table` | 3,976 |

The current `release-small` profile already has the right stable defaults for
this goal: `opt-level = "z"`, one codegen unit, fat LTO, aborting panics, and
stripped symbols. A minimal Rust `std` hello-world program built locally with
the same relevant settings was 285,136 bytes. That is not a universal lower
bound, but it shows that roughly 285 KiB of this build is toolchain/runtime/ELF
floor rather than glam functionality.

An unstripped measurement attributed named symbol bytes approximately as
follows. These figures are directional: monomorphized parser code often carries
the `g_syntax` name even when Chumsky supplies most of the machinery.

| Area | Approx. symbol bytes | Share of named bytes |
| --- | ---: | ---: |
| standard-library backtrace machinery | 170,823 | 25.5% |
| `g_syntax` and parser instantiations | 155,848 | 23.3% |
| `eval` | 97,035 | 14.5% |
| other `std`/`core`/`alloc` | 62,329 | 9.3% |
| `interaction_net` | 38,575 | 5.8% |
| bigint/rational/integer crates | 34,675 | 5.2% |
| `core` | 23,919 | 3.6% |

The useful conclusions are:

1. The interaction-net representation is not disproportionately large.
2. Parser combinator monomorphization and evaluator/builtin dispatch are the
   first application-owned places to experiment.
3. Exact rational arithmetic costs real space, but approximately 35 KiB of
   named code is modest for a deliberate language semantic.
4. Stable `panic = "abort"` does not remove all precompiled standard-library
   backtrace/unwind support. Eliminating that generally means a nightly/custom
   `build-std` or a substantially different runtime target, not another normal
   profile flag.

Measured configuration experiments:

| Experiment | Size | Difference | Assessment |
| --- | ---: | ---: | --- |
| current profile | 903,112 | — | baseline |
| disable forced unwind tables | 903,112 | 0 | no benefit |
| change `opt-level` from `z` to `s` | 1,005,368 | +102,256 | current `z` choice is substantially better |
| Chumsky without default `stacker` | 899,256 | -3,856 | too little benefit for lost deep-recursion protection |
| non-PIE executable | 883,416 | -19,696 | poor default trade for ASLR/security |

Nightly's immediate-abort panic mode was not measurable on the installed stable
toolchain. It and a custom standard library may be worth revisiting only if a
strict sub-700-KiB deployment budget becomes a product requirement.

Recommended size work, in order:

1. Add a reproducible size report or budget around
   `target/release-small/glam`; record bytes and the largest sections. This
   prevents accidental regressions and makes every later experiment honest.
2. Remove the dead compatibility surface listed above and consolidate repeated
   CLI/diagnostic/lazy-value paths. Expect clarity gains; measure rather than
   promising a large binary reduction.
3. Split `apply_builtin` by semantic family, then compare both size and runtime.
   Centralize its repeated fixed-arity extraction/error path at the same time.
   The split is justified by clarity even if LTO produces the same code.
4. Experiment with type-erasing (`.boxed()`) a few stable Chumsky parser
   boundaries that are reused in many branches. Boxing trades dispatch and
   allocation for less generic instantiation, so accept it only with a measured
   size win and parser benchmark/smoke-test parity.
5. If size remains important, use a dedicated analysis tool such as
   `cargo-bloat` in development/CI. Do not add it as a runtime dependency.

With the current `std`, parser, exact-number, and evaluator feature set, reducing
the executable by tens of KiB is realistic; approximately 800–850 KiB is a
reasonable experimental target, not a promised budget. Getting materially
below 700 KiB likely requires changing parser/runtime capabilities or building
a custom standard library. SectorLisp-scale output would require an altogether
different runtime and dependency strategy, not ordinary Rust cleanup.

## Performance observations

There is currently no benchmark harness. Before changing runtime containers,
record three representative workloads: parsing/lowering a nontrivial source,
reducing a cursor-heavy shared net, and evaluating builtin/list/object paths.
Wall time plus allocations or reduction counts are sufficient initially; a
large benchmarking framework is not required.

The following are low-risk candidates to measure after the clarity cleanup:

- Cache frequently used interned core keys (`apply`, `eff`, `spec`, `name`,
  `deps`, `defs`, and similar) rather than rebuilding bytes and consulting the
  interner at every use. A small explicit key set is clearer than scattered
  constructor calls and should reduce allocation/hash work.
- `BindingId` is allocated densely within one resolver context, but lowering
  uses ordered maps/sets for uses and captures. Where deterministic ordering is
  not semantically required, a context-local indexed vector or bitset could
  replace tree lookups. Preserve deterministic capture order explicitly.
- Runtime node IDs, ports, copy IDs, and fan sites are trusted integer-like
  keys, yet their hash tables use the standard randomized hasher. Benchmark a
  small deterministic integer hasher for runtime-local maps. This may improve
  reduction throughput, but it can affect executable size and must not be used
  for attacker-controlled keys.
- Avoid the duplicate source `String`/`Arc<[u8]>` ownership described above.
  This is a straightforward reduction in parsing peak memory and copying.
- Staged operators clone all previously supplied values into a new allocation
  for each input. This is negligible for today's mostly unary/binary builtins,
  but is quadratic in a function's capture count or a large literal's arity.
  The proposed `StagedArgs` consolidation should preserve the current simple
  representation first; then benchmark a persistent or uniquely owned
  accumulator if high-arity sources make this visible.
- Where diagnostic-only methods such as `blocked_cursors` build a fresh map,
  expose an iterator or write directly into the diagnostic if profiles show the
  allocation matters. Do not complicate the runtime merely to optimize an
  exceptional observation path.
- The strict and forcing variants of list traversal share substantial logic.
  Consider one internal traversal kernel/policy only if it remains readable;
  generic abstraction can increase monomorphized code, so compare source
  simplicity, binary size, and list benchmarks together.

Deliberately defer scheduler replacement, parallel evaluation, node-ID reuse,
arena storage, and reachability-style GC. The current ordered active-pair tree
encodes useful exact state and matches the chosen semantics. No evidence from
this review supports changing it before profiling real programs.

## Prioritized recommendations

### Do next

1. Remove or justify the four unused/redundant APIs and fix the checked exponent
   conversion.
2. Reduce `AgentContext.md` to invariants and links; make `src/README.md` the
   concise current module/dataflow map; archive or rewrite the historical object
   spike.
3. Perform the focused lifecycle and lambda-era renames, especially lazy-value
   operations, `term`, and the stale lowering/fixpoint comments.
4. Start decomposing `g_syntax.rs` at the existing resolved/lowering boundary,
   one behavior-preserving move at a time.
5. Add a reproducible release-small byte/section report before attempting size
   optimizations.

### Then measure

6. Split builtin evaluation and shared-net driving out of `eval.rs`; compare the
   resulting size to the recorded baseline.
7. Try parser boxing at a small number of high-fan-out boundaries.
8. Cache common core keys and benchmark integer-key runtime hashers.
9. Consolidate source ownership and, if still worthwhile, list traversal
   internals.

### Defer

- Nightly/custom-standard-library builds, `no_std`, non-PIE output, removal of
  parser stack protection, and semantic feature cuts solely for size.
- Scheduler, allocation, GC, and parallel-runtime redesigns.
- Broad lint-driven renaming or enabling all pedantic Clippy lints. Default
  Clippy currently reports mostly local style issues; use context structures to
  resolve genuinely over-parameterized lowering functions rather than editing
  code solely to silence warnings.
