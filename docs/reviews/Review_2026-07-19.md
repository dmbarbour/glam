# Vestigial-Code and Simplification Review — 2026-07-19

## Scope

This is a read-only review of the current Rust bootstrap, focused on code and
configuration that can be removed or made materially simpler without changing
intended semantics. It follows the large architectural cleanup recorded in
`Review_2026-07-16.md`; findings from that review are repeated only when the
current implementation still contains the issue.

The review distinguishes:

- **remove now**: no current caller, obsolete compatibility, or redundant API;
- **simplify next**: current behavior is needed, but its representation or
  ownership can be reduced safely;
- **retain deliberately**: apparently unusual code that still protects an
  active invariant or an explicitly deferred design.

No implementation source is edited as part of this review.

## Method

- Read the current implementation map, agent-context invariants, and focused
  architecture notes before treating anything as vestigial.
- Compare public exports and `pub` items with in-tree callers.
- Audit dead-code allowances, transitional comments, compatibility branches,
  duplicated state machines, wrappers, and facade re-exports.
- Trace candidates through tests before recommending deletion.
- Run the normal formatter, tests, and Clippy as a clean baseline; use search
  results as leads rather than proof.

## Executive summary

The bootstrap is in substantially better shape than it was at the prior
review. The old expression evaluator is gone, the large modules have useful
internal boundaries, every direct dependency has an identifiable current
owner, and the formatter, tests, and Clippy all pass. I did not find an entire
obsolete subsystem that should be deleted.

The remaining vestigial code is mostly *surface area left behind by spikes*:

- scheduler and interaction-net inspection helpers used only by tests are
  still compiled as production APIs;
- two composed reflection runners have no callers;
- the interaction-net facade suppresses unused-import warnings around an
  over-broad set of re-exports;
- `compiler`, `g_syntax`, and `Builtin` remain public primarily so `main` and
  integration tests can bypass the intended library facade;
- completed evaluation queries notify a condition variable even though query
  results deliberately are not scheduler wake conditions.

The most valuable implementation simplifications are to give `LazySource` one
explicit dispatch match, share the transactional reflection-state mechanics
used by assembler and logger hosts, and stop rebuilding an evaluation session
when only the assembler's file host changes. These changes delete branching
and duplicated policy rather than deleting semantics.

Most of the removals below will primarily improve clarity and compile-time
surface. LLVM probably already removes many uncalled private-crate functions
from `release-small`, so they should not be presented as a large executable
size win without measuring afterward.

## Findings

### 1. Move test-only inspection helpers out of production builds (DONE)

**Category:** remove now

Several methods are reachable only from `#[cfg(test)]` code but are exposed as
ordinary `pub` or `pub(crate)` methods. Two of them carry explicit
`allow(dead_code)` attributes:

- `EvaluationTaskHandle::wait` and `EvalContext::with_new_task` in
  `evaluation.rs`;
- `EvalContext::{reflection_task_count, evaluation_query_count,
  shares_session_with}`;
- `Value::get_atom_path` and the unused `LazyValue::label` accessor in
  `core.rs`;
- `InteractionNet::{nodes, wires, active_pairs}` in
  `interaction_net/model.rs`; all current callers are topology tests, including
  the g-syntax lowering tests;
- `CopyId::get`, for which there is no caller at all.

Keep the behavior needed by tests, but put test inspection behind `#[cfg(test)]`
or in a test-support extension trait. Delete accessors with no caller. This
also lets the two production `allow(dead_code)` attributes disappear.

The nested `Associativity` enum in `g_syntax/parser/expression.rs` currently
constructs all three variants, so its `allow(dead_code)` appears obsolete and
should be removed. The size regression test in `core.rs` is a different case:
its comparison enum intentionally is not executed. Retain an allowance there,
but rename its stale `Pending` variant to `Promised` and make the comparison
enum reflect all current pointer-sized lazy sources.

### 2. Narrow the interaction-net facade instead of suppressing warnings (DONE)

**Category:** remove now

`interaction_net.rs` applies `allow(unused_imports)` to all three facade
re-export groups. The facade currently republishes construction conveniences,
model internals, runtime diagnostics, and cursor state whether or not sibling
modules use them.

The root module is private, so this is not a compatibility requirement for
library clients. Replace the wildcard-like facade with explicit
`pub(crate) use` declarations for types used outside `interaction_net`, and
have interaction-net submodules import directly from `model`, `builder`, or
`runtime`. Tests that need `Node`, `RuntimeNet`, or residual identities can
import their defining module. The goal is for an unused re-export to become a
normal compiler warning again.

Do not make the generic interaction-net implementation public as part of this
cleanup. The checked, opaque `api::NetBuilder` remains the intended embedding
surface.

### 3. Remove unused composed reflection entry points (DONE)

**Category:** remove now

`reflection.rs` exposes four closely related runners:

- `run`;
- `run_with_reflection_host`;
- `run_unit_with_reflection_host`;
- `run_unit_with_reflection_host_in_runtime`.

Only the runtime-attached unit runner has a production caller (`main`), while
the first runner underlies `run_standard` and focused tests. The two middle
composed wrappers have no in-tree caller. Because the project is pre-release,
delete those two rather than carrying speculative public combinations.

Longer term, prefer one composed-task constructor plus explicit policies such
as `requiring_unit_result()` and “attach to this runtime.” This avoids adding a
new free function for every combination of child capabilities, result policy,
and executor ownership.

### 4. Stop waking the task scheduler for immutable query completion (COURSE CORRECTED)

**Category:** remove now

`EvalContext::complete_query` and `cancel_reserved_query` both notify
`EvaluationSession::task_changed`. No scheduler path waits on a query ID:
`.query_result` polls the query table and returns effect failure while the
transactional query request is uncommitted. Once committed, the query snapshot
is completed synchronously and is immutable by design.

Remove those two notifications. If asynchronous external queries are added
later, introduce an explicit query wait token then; do not make the general
task condition variable imply wake semantics the API does not yet have.

COURSE CORRECTION: queries use a private volume within the session for the result.
This accounts for refl annotations and sharing.

### 5. Close the transitional Rust front-end surface (DONE)

**Category:** simplify next

`lib.rs` still says `compiler` and `g_syntax` are public for `--parse` “until
reflection is implemented,” even though reflection tasks now exist. The
resulting public surface exposes the complete syntax AST, a `CompileContext`
whose useful values are otherwise private implementation types, and
`lower_to_core_with_context`. Integration tests use that escape hatch, and
`main` needs only `parse_source`, `ParsedSource`, and `DeclarationKind` for its
parse summary.

This exposure also forces `core::Builtin` into the public facade. Outside the
front end, `main` is its only production client, using two variants solely to
construct the default empty `conf.env` object. Thus a large evaluator enum and
`Value::{builtin,builtin_call}` are public to serve one narrow bootstrap
operation.

Add a small library operation for built-in g-source inspection (or a
CLI-private adapter returning a parse summary), then make `compiler` and
`g_syntax` private. Move the invalid-sample assertions onto that inspection
surface or into crate tests. Provide a narrow opaque helper for the empty
environment object rather than exposing object implementation builtins. After
that, remove `pub use core::Builtin` and make the two public `Value` builtin
constructors crate-private.

This matches the intended architecture: a user-defined syntax is eventually a
Glam value, not a Rust client of the bootstrap's syntax AST or core builtin
enum.

### 6. Make `CompileContext` a capability context, not a constructor bag (DONE)

**Category:** simplify after finding 5

Once the transitional public module is closed, the thin constructor methods
`value_number`, `value_binary`, `value_atom`, `value_dict`, `value_builtin`,
and `empty_dict_value` add little abstraction. Their use is already
inconsistent: g-syntax sometimes calls them and sometimes constructs
`Value::Builtin` directly.

Retain the operations that represent assembler capabilities or authority:
relative imports, diagnostic emission, `abstract_global_path`, prior/final
definitions, and the canonical unit value. Let the built-in front end create
ordinary internal data directly. This makes it much easier to see which
operations a future user-defined compiler must request from the assembler and
which are merely Rust representation details.

At the same time, rename `lower_to_core_with_context`. There is no longer a
`CoreExpr` backend; it resolves g syntax and emits closed semantic values/nets.
`lower_source` or `lower_module_source` describes the current boundary without
reviving the removed expression architecture.

### 7. Match `LazySource` once and centralize post-force policy

**Category:** simplify next

`eval/value.rs::eval_lazy` currently asks a `LazyValue` separately for
`net_computation`, `function_call`, `builtin`, `reflection_gate`, both fixpoint
forms, deferred state, access state, and whether it is promised. Each query is
a separate match hidden behind a one-variant accessor in `core.rs`.

The same function also computes two implicit policies before forcing:

- reflection annotations must continue through a newly produced gate;
- net computations, function calls, and a hand-maintained subset of builtins
  must force the result again.

That builtin list is easy to forget when another staged builtin is introduced,
and cached results duplicate the same post-processing decision before and
after the lazy claim.

Make `LazySource` crate-visible and let the evaluator perform one exhaustive
match. Associate each source with an explicit post-force policy such as
`Return`, `ForceResult`, or `RunReflectionGate`, and put cached-result handling
in one helper. Keep the source representation in `core`; only its operational
interpretation belongs in `eval`. This removes seven projection methods and
makes adding a source or staged builtin an exhaustiveness-checked edit.

### 8. Do not rebuild evaluation state when only the file host changes (DONE)

**Category:** remove now

`Assembler::with_host` replaces the external file capability and then creates
a fresh `EvaluationSession`. The host is not captured by that session; the
session depends on the reflection environment, diagnostic sink, and executor.
The rebuild is therefore redundant and can unexpectedly discard scheduled
reasoning if a caller treats the fluent method as an ordinary late update.

Have `with_host` replace only `self.host`. For the three settings that really
do affect evaluation (`diagnostic_sink`, reflection environment, and worker
executor), introduce one private `rebuild_evaluation_session` helper and state
clearly that changing them starts a fresh session. This removes repeated
constructor blocks and makes session invalidation explicit.

### 9. Share transactional reflection-state mechanics between hosts

**Category:** simplify next

`AssemblerReflectionHost` in `api.rs` and both `TaskHost` implementations for
`LoggerTaskHost` in `main.rs` repeat the same core protocol:

1. lock generation plus heap;
2. compare the transaction generation;
3. replace the heap;
4. increment the generation and notify waiters;
5. release the lock, emit diagnostics, and commit deferred reflection updates.

The logger's main specialization legitimately adds input consumption and
transactional stderr, so the host types should not be collapsed. Extract only
the reusable state/protocol, for example a `ReflectionState { generation,
heap }` plus a small checked-commit helper. Let each host own its extra snapshot
and journal work around that helper.

This is preferable to a large generic host abstraction: it removes the exact
duplication most likely to drift while preserving the specialization boundary
that keeps logger-only effects out of child reflection tasks.

### 10. Finish the current reflection module boundary

**Category:** organization after behavioral cleanup

`reflection.rs` remains 4,191 lines, of which about 2,750 are production code.
`requests.rs` successfully separated reusable reflection requests, but the
root still contains three distinct concepts:

- public specialization/host types and run entry points;
- the `EffectTask` cooperative interpreter and branch scheduler;
- continuation, reset-stack, fixpoint, and transaction machinery.

After findings 3 and 9 reduce the cross-cutting surface, move the machine into
`reflection/task.rs` and the continuation/state encoding into
`reflection/control.rs`. Keep `reflection.rs` as the public facade and
composition layer. This is a navigation improvement, not an excuse to change
effect semantics or duplicate generic types.

### 11. Remove chronological terminology left by earlier spikes

**Category:** small clarity cleanup

Several comments and names describe history rather than current behavior:

- `BindingId` contrasts itself with a removed “Core compatibility backend”;
- `ResolvedExpr::apply` promises work to a “future interaction-net emitter”
  that now exists;
- `CompileContext::final_defs` says only “future dictionary” instead of
  “promised final definitions”;
- parser diagnostics say unsupported forms are unavailable “in the current
  spike” without identifying any actual future contract;
- the `lib.rs` note ties front-end visibility to reflection not existing;
- several g-syntax tests still say they lower to “core terms,” although they
  now assert semantic values and closed nets.

Update these while renaming `lower_to_core_with_context`. Consider renaming
`FunctionCode` to `FunctionNetTemplate` or another name that identifies its
shared runtime-net role; “code” alone obscures the fact that lowering has
already happened.

## Elements that should not be removed yet

- `EvaluationTaskState::Dormant` and `Reserved` are not duplicates. Dormant
  records back promise/fixpoint waits; reserved records represent
  transactionally requested tasks that have not committed.
- The internal list-effect, object, and effect builtins all have live
  constructors and evaluator paths. They may eventually move into Glam code,
  but none is dead today.
- `MachineStep`'s large-enum Clippy allowance has a local explanation and
  avoids boxing every common cooperative transition. Keep it unless profiling
  changes the tradeoff.
- The simple `Diagnostic` projections duplicate information available through
  the rich emission value, but they are still the usable compatibility surface
  for the default logger and embedding tests. Remove them only together with a
  replacement public inspection/rendering API.
- Claimed, blocked-call, blocked-cursor, and stuck active-pair states all have
  live scheduler or diagnostic roles. Their similarity is not evidence that
  they should be collapsed.
- All ten direct Cargo dependencies have current code owners. No dependency
  should be removed based on this pass.

## Recommended sequence

1. Delete or test-gate the inspection helpers in finding 1, narrow the
   interaction-net facade, remove the query notifications, and stop rebuilding
   sessions in `with_host`. These are small, independently testable edits.
2. Remove the two unused reflection runner combinations and introduce the
   shared session-rebuild helper.
3. Add the narrow parse-inspection and empty-object facilities needed by
   `main`, then privatize `compiler`/`g_syntax` and hide `Builtin`.
4. Reduce `CompileContext` to capabilities and rename the obsolete core
   lowering terminology.
5. Refactor `eval_lazy` around one exhaustive `LazySource` match, preserving
   the existing cached, blocked, and recursive-fixpoint tests.
6. Extract the small transactional reflection-state helper, then split the
   effect machine/control implementation into focused submodules.

Run the ordinary formatter, full tests, and Clippy after every slice. The
front-end visibility cleanup should also run the CLI parse tests and invalid
sample tests; the lazy-source cleanup should run the evaluator and reflection
tests specifically before the full suite.
